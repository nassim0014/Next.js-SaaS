# Architecture — AI SaaS Boilerplate

> Phase 2 design artifact. Folder structure, data-flow, and design rationale.
> All paths are relative to project root.

---

## 1. Tech Stack (Locked)

| Layer | Choice | Why |
|---|---|---|
| Framework | **Next.js 16+ (App Router)** | RSC, streaming, server actions, partial prerendering |
| Language | **TypeScript (strict)** | Zero `any` policy enforced via ESLint |
| DB | **PostgreSQL (Supabase)** + pgvector | One engine for relational + vector + auth + storage |
| ORM | **Prisma 5+** | Type-safe queries, migrations, schema-as-code |
| Auth | **Supabase Auth** + RBAC layer | Email/magic link/OAuth, JWT in cookies |
| UI | **shadcn/ui + Tailwind v4** | Copy-paste components, no lock-in |
| Validation | **Zod** | Shared between client, server actions, and Prisma |
| AI | **Vercel AI SDK** | Streaming, tool calls, multi-provider |
| Billing | **Stripe** (primary) + **Lemon Squeezy** (alt) | Merchant-of-record option for EU |
| Testing | **Vitest** (unit) + **Playwright** (e2e) | Modern, fast, App-Router-aware |
| Observability | OpenTelemetry → Posthog / Sentry hooks | Cost + error + product analytics |

## 3. Request Lifecycle (Auth + RBAC + Audit)

```
Client
  │
  ▼
middleware.ts  ──── refresh Supabase session ──── route guard (marketing/auth/app)
  │
  ▼
(app)/layout.tsx ── resolve active org from subdomain/cookie
  │
  ▼
Page / Server Action
  │
  ├── lib/auth/session.ts ──── requireUser()
  ├── lib/auth/org-context.ts ─ getActiveOrg()
  ├── lib/auth/rbac.ts ──────── can(user, "conversations:create", org)  ❌ → 403
  ├── lib/prisma.ts ─────────── transactional query
  ├── lib/audit/logger.ts ───── audit(LOG, ...)  (async, fire-and-forget)
  │
  ▼
Response (streamed if /api/chat)
```

## 4. Multi-Tenancy Model

**Decision: shared DB with `organizationId` on every tenant-scoped table.**

Rationale:
- Cheaper than schema-per-tenant at scale (Supabase free tier caps schemas).
- One migration path, one backup, one place to query for analytics.
- RLS policies on Supabase enforce row isolation at the DB layer (defense-in-depth, even if app layer has a bug).

Trade-off: noisy-neighbor risk on the shared DB → mitigated by rate-limits at the Edge Function layer + per-org query budgets.

Every tenant-scoped Prisma model includes:
- `organizationId String @db.Uuid`
- `@@index([organizationId, <filterable column>])`
- A cascading FK to `Organization`

## 5. AI Cost Observability — The ⭐ 5th USP

Every LLM call flows through `lib/ai/cost.ts`:

```
streamChat({ orgId, userId, conversationId, model, messages })
  ├── 1. resolve ModelConfig → pricing
  ├── 2. check org budget cap (UsageRecord + Plan.tokenQuota) → throw if exceeded
  ├── 3. call provider (Vercel AI SDK)
  ├── 4. on every chunk: accumulate tokens
  ├── 5. on completion: insert TokenUsage row (async)
  └── 6. update Conversation.tokenCount + estimatedCostUsd (debounced)
```

This powers:
- **Live cost dashboard** at `/usage` (per-day, per-agent, per-user breakdowns)
- **Budget caps** — when an org hits 80% / 100% of plan quota, fire email + webhook
- **Usage-based billing** — overage priced per 1K tokens, reconciled via Stripe metered billing
- **Per-conversation cost attribution** — show users "this chat cost $0.04"

No AI SaaS boilerplate ships this. Founders get burned by token bills because they can't see them. We solve that on day one.
